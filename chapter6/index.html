Resilient Web Design

Chapter 6: Etapes

“Il faut toujours concevoir une chose en envisageant l’amélioration possible”, a déclaré l'architecte finlandais Eliel Saarinen. “Une chaise dans une pièce, une pièce dans une maison, une maison dans un environnement, un environnement dans un plan de ville.”

À première vue, le web design semble être une sorte de design graphique. L'utilisation d'outils de conception graphique comme Photoshop pour concevoir des sites Web renforce cette vision. Mais pour aller au cœur de l'objectif d'un site Web, il faut considérer l'interface dans son contexte plus large : qu'est-ce que les gens essaient d'accomplir ?

Quand tu design pour le web c’est tentant de penser qu’en moyens nécessitant des interactions telles que taper, cliquer, faire défiler, glisser-déposer. Mais très peu de gens se réveillent le matin en se disant qu’ils allaient juste faire des tapotements ou des défilements sur leurs téléphones. Ils sont plus susceptibles de penser en termes de lecture, d'écriture, de partage, d'achat et de vente. Les Web designers doivent voir au-delà des actions que l’utilisateur effectue mais faire en sorte de trouver les choses les plus significatives pour l’utilisateur.

Dans ce livre Designing With Progressive Enhancement, the Filament Group définit cette technique et l’appel “ la perspective des rayons x

Prendre une perspective radiographique, c'est regarder "à travers" les widgets complexes et les styles visuels d'un design, identifier le contenu principal et les éléments fonctionnels qui composent la page, et trouver un simple équivalent HTML pour chacun d'eux qui fonctionnera de façon universelle. ”

Si vous n'êtes pas habitué à cette approche de la conception Web, cela peut prendre du temps pour s'y habituer. Au bout d’un au bout d’un certain temps il est facile de s’y habituer et après il devient compliqué de ne pas juger un site. C'est comme essayer de ne pas voir la flèche dans l'espace blanc du logo FedEx, ou essayer de ne pas se rappeler que tous les canards portent en fait un masque de chien.

Voici une approche en trois étapes que j’adopte pour la conception de sites web :

  1. Identifier les fonctionnalités de base .
  2. Utiliser la technologie la plus simple possible pour la rendre fonctionnel.
  3. Améliorer!

Identifier les fonctionnalités de base peut sembler assez simple, et ça devient le cas après un peu de pratique. Mais il peut être difficile au début de séparer ce qui est vraiment nécessaire de ce qui est agréable à avoir.

Information

Disons que vous êtes un fournisseur de nouvelles. C'est là que se trouve la fonctionnalité de base, à savoir fournir des nouvelles. Il y a beaucoup, beaucoup d'autres services que vous pourriez également fournir ; puzzles interactifs, notifications en temps réel, et plus encore. Aussi précieux que soient ces services, ils ne sont probablement pas aussi importants que de s'assurer que les gens aient accès aux nouvelles.

Une fois cette fonctionnalité de base identifiée, il est temps de passer à la deuxième étape : comment pouvez-vous rendre cette fonctionnalité de base disponible en utilisant la technologie la plus simple ?

Théoriquement, un fichier texte serait le moyen le plus simple de fournir les nouvelles. Mais comme il s'agit spécifiquement du web, pour rendre les fonctionnalités de bases disponibles en utilisant la technologie web la plus simple, la meilleur méthode serait d’utiliser un fichier HTML avec une URL.

Même à ce stade précoce, il est possible de compliquer les choses pour rien. Le HTML pourrait être inutilement gonflé. L'URL pourrait être compliquée, difficile à partager ou difficile de retenir.

Maintenant que les news ont été balisées avec les éléments HTML appropriés - articles, titres, paragraphes, listes et images - il est temps de passer à la troisième étape : améliorer !

Par défaut, les news seront présentées à l'aide de la feuille de style du navigateur. C'est lisible, mais pas vraiment agréable. En appliquant votre propre CSS , vous pouvez sculpter le contenu dans une forme plus agréable. Espaces blancs, plomb, couleurs et contrastes sont à votre disposition. Vous pouvez même utiliser des polices personnalisées - une amélioration qui a été impossible sur le Web pendant de nombreuses années.

Vous pouvez même utiliser des polices personnalisées - une amélioration qui a été impossible sur le Web pendant de nombreuses années. Il n’y a aucune garantie que les déclarations CSS puisse être lu par tous les navigateurs. C'est pas grave. Ces navigateurs ignorent ce qu'ils ne comprennent pas. Et surtout, les nouvelles sont toujours accessibles à tous, quelles que soient les capacités CSS de leur navigateur.

Pour les navigateurs sur les appareils à grand écran, vous pouvez introduire la mise en page. Il peut sembler étrange au premier abord de considérer la mise en page comme une amélioration, mais c'est la leçon d'un design mobile‐first responsive . Considérez d'abord le contenu, puis marquez-le avec un ordre de source raisonnable, puis appliquez les déclarations de mise en page dans les requêtes média.

Grâce à la constante évolution du CSS, il existe de multiples façons d'appliquer la mise en page. Comme Andy Tanenbaum l'a dit:

The nice thing about standards is that you have so many to choose from.

Communication

L'application du processus en trois étapes à un site de news est relativement simple. Poster l'actualité est un acte assez passif. Pour vraiment tester ce processus, nous devons l'appliquer à quelque chose de plus interactif.

Supposons que nous construisions un réseau social. Les personnes qui utilisent notre outil doivent pouvoir communiquer entre elles, peu importe où elles se trouvent dans le monde. La fonctionnalité de base est l'envoi et la réception de messages.

L'affichage des messages dans un navigateur Web n'est pas difficile. Il peut y avoir beaucoup de complexité sur le serveur impliquant des bases de données, la synchronisation, la mise en file d'attente et l'équilibrage de charge, mais le HTML nécessaire pour structurer une liste chronologique n'est pas très différent du HTML nécessaire pour un site de nouvelles.

L'envoi d'un message depuis le navigateur vers le serveur web nécessite un HTML interactif. C'est là que les formulaires entrent en jeu. Dans ce cas, un formulaire avec une saisie de texte et un bouton “submit”’ devrait suffire, au moins pour la fonctionnalité de base.

Les gens peuvent maintenant recevoir et répondre aux messages sur notre réseau social, peu importe le type d'appareil ou de navigateur qu'ils utilisent. L'astuce consiste maintenant d’améliorer l'expérience sans briser cette activité fondamentale.

Si nous laissions le site dans cet état uniquement HTML, je ne pense pas que nous célébrerons l'introduction en bourse de notre société dans un avenir proche. Pour vraiment nous démarquer de la concurrence, nous avons besoin de cette troisième étape du processus : améliorer !

À tout le moins, nous pouvons appliquer la même logique que nous avons utilisée pour le site de nouvelles et le style de notre service. En utilisant CSS, nous pouvons fournir la couleur, la texture, le contraste, les polices Web et, pour les grands écrans, la mise en page. Mais ne nous arrêtons pas à la présentation. Améliorons aussi l'interaction.

À l'heure actuelle, ce réseau social a le même type d'interaction basée sur les pages qu'un site de nouvelles. Chaque fois que quelqu'un envoie un message au serveur, le serveur renvoie une nouvelle page entière au navigateur. Nous pouvons faire mieux que cela. C'est l'heure de l'Ajax.

Nous pouvons intercepter la soumission du formulaire et envoyer les données au serveur en utilisant Ajax-I comme le mot Hijax pour décrire ce type d'interception Ajax. S'il y a une réponse du serveur, nous pouvons aussi mettre à jour une partie de la page en cours au lieu de rafraîchir la page entière. Ce serait aussi un bon moment pour présenter une animation appropriée.

Nous pouvons aller plus loin. Les navigateurs qui prennent en charge les WebSockets peuvent recevoir des messages du serveur. Les personnes qui utilisent ces navigateurs pourraient recevoir des mises à jour dès qu'elles ont été envoyées. Il est même possible d'utiliser des connexions peer-to-peer entre navigateurs pour permettre aux gens de communiquer directement.

Tous les navigateurs ne prennent pas en charge cette fonctionnalité avancée. C'est pas grave. La fonctionnalité de base - envoyer et recevoir des messages - est toujours accessible à tous.

Creation

What if our social network were more specialised? Let’s make it a photo‐sharing service. That raises the bar a bit. Instead of sending and receiving messages, the core functionality is now sending and receiving images.

The interface needs to show a reverse‐chronological list of images. HTML can handle that. Once again we need a form to send data to the server, but this time it needs to be a file upload instead of a text field.

With those changes, the core functionality is in place. Time to enhance.

As well as all the existing enhancements—CSS, web fonts, Ajax, WebSockets—we could make use of the File API introduced in HTML5. This allows us to manipulate the image directly in the browser. We could apply effects to the image before sending it to the server. Using CSS filters, we can offer a range of image enhancements from sepia tones to vignettes. But if a browser doesn’t support the File API or CSS filters, people can still upload their duck‐facing selfies.

Collaboration

There was a time when using software meant installing separate programs on your computer. Today it’s possible to have a machine with nothing more than a web browser installed on it. Writing emails, looking up contact details, making calendar appointments, bookkeeping and other financial tasks can all be done without having to install bespoke applications. Instead, the act of visiting a URL can conjure up the tool you need when you need it.

Delivering software over the web doesn’t just replace the desktop‐centric way of working. The presence of an internet connection opens up possibilities for all kinds of collaboration. Take, for example, the kinds of applications that were once called “word processors.” As long as those programs were tethered to individual machines, trying to collaboratively edit a document was bound to be a tricky task requiring plenty of coordination, sending files back and forth. Using the web, the act of sharing a single URL could allow multiple people to work on the same document.

Let’s apply the three‐step process to a web‐based word processor:

Identify core functionality.

The tautological answer would be “processing words.” Not very helpful. What do people actually do with this software? They write. They share. They edit.

Make that functionality available using the simplest possible technology.

Looking at our three verbs—writing, sharing, and editing—we get one of them for free just by using URLs: sharing. The other two—writing and editing—require the use of a form. A basic TEXTAREA element can act as the receptacle for the words, sentences, and paragraphs that will make up everything from technical reports to the great American novel. Submitting that content to a web server means it can be saved for later.

Technically, that’s a web‐based word processor, accessible to anyone with a web browser and an internet connection. But the experience is clunky and dull. It would be a shame not to take advantage of some of slicker options available in modern browsers.

Enhance!

Using JavaScript, the humble TEXTAREA can be replaced with a richer editing interface, detecting each keystroke and applying styling on the fly. Web fonts can make the writing experience more beautiful. Ajax will allow work to be saved to the server almost constantly, without the need for a form submission. WebSockets provide the means for multiple people to work on the same document at the same time.

Both Ajax and WebSockets require an internet connection to work. There’s no guarantee of a stable internet connection, especially if you’re trying to work on a train or in a hotel. Modern browsers provide features which, after the initial page load, can turn the network itself into an enhancement.

If a browser supports some form of local storage, then data can be stored in a client‐side database. Flaky network connections or unexpected power outages won’t get in the way of saving that important document. Using Service Workers, web developers can provide instructions on what to do when the browser (or the server) is offline.

These are modern browser features that we should be taking full advantage of …once we’ve made sure that we’re providing a basic experience for everyone.

Scale

Ward Cunningham, the creator of the wiki, coined the term “technical debt” to describe a common problem in the world of software. Decisions made in haste at the beginning of a project lead to a cascade of issues further down the line. I like to think of the three‐step layered approach as a kind of “technical credit.” Taking the time to provide core functionality at the beginning gives you the freedom to go wild with experimentation from then on.

Some people have misunderstood progressive enhancement to mean foregoing the latest and greatest browser technologies, but in fact the opposite is true. Taking a layered approach to building on the web gives you permission to try cutting‐edge JavaScript APIs, regardless of how many or how few browsers currently implement them.

We’ve looked at some examples of applying the three‐step approach to a few products and services—news, social networking, photo sharing, and word processing. You can apply this approach to many more services: making and updating items in a to‐do list, managing calendar appointments, looking up directions, making reservations at nearby restaurants. Each one can be built with the same process:

  1. Identify core functionality.
  2. Make that functionality available using the simplest possible technology.
  3. Enhance!

This approach works at different scales. It doesn’t just work at the highest level of the service; it can also be applied at the level of individual URLs within.

Ask “what is the core functionality of this URL?”, make that functionality available using the simplest possible technology, and then enhance from there. This can really clarify which content is most important, something that’s important in a mobile‐first responsive workflow. Once you’ve established that, make sure that content is sent from the server as HTML (the simplest possible technology). Then, using conditional loading, you could decide to make Ajax requests for supporting content if the screen real‐estate is available. For the URL of an individual news story, the story itself would be sent in the initial response, but related stories or comments could be pushed from the server only as needed (although you can still provide links to the related stories and comments for everyone).

We can go deeper. We can apply the three‐step process at the scale of individual components within a page. “What is the core functionality of this component? How can I make that functionality available using the simplest possible technology? Now how can I enhance it?”

A component might be designed to be an all‐singing all‐dancing interactive map. With x‐ray goggles, the core functionality reveals itself to be something much simpler: showing a location. Provide the address of that location in text: the simplest possible technology. Now you can enhance.

It’s worth remembering that enhancements can be provided on a sliding scale. The first enhancement for a text address might be to provide a static image. The next level up from that would be to swap out the static image with an interactive Ajax‐powered slippy map. If a browser supports the geolocation API, you could show the distance to the location. Layer on some animations and transitions to help convey the directions better.

Site navigation is another discrete component that lends itself well to a sliding scale of enhancements. The core functionality of navigation is to provide links to resources. The simplest—and still the best—technology to enable that is the humble hyperlink. A list of links should do the trick. With that in place, you are now free to enhance it into something really compelling. Off‐canvas navigation, progressive disclosure, sliding, swiping, fading, expanding …the sky’s the limit.

Because enhancements can be layered on according to the capabilities of each browser, it quickly becomes clear that this approach doesn’t simply reduce down to having two versions of everything (the basic version and the enhanced version). Instead the service, the URLs, and the components you are designing could be experienced in any number of ways. And that’s okay.

Websites do not need to look exactly the same in every browser.
